## Mandatory depth rules

These rules are non-negotiable and apply to every step above. They do not change
the file set, the step sequencing, or the output filenames. They raise the floor
on how deep each step must go when a family is directly surfaced in the PDF.

Definition — "directly surfaced":
A family (item, sub-item, table, note family, exhibit, roster, endorsement,
addendum) is directly surfaced if it physically appears in the PDF body,
exhibits, or addenda — regardless of whether it was easy or hard to read.

### Rule 1 — No "light" coverage for directly surfaced high-value families
A1 may NOT mark any directly surfaced item, table, note family, or exhibit as
"light", "summary-only", "labeled", or "headline-only" coverage.
For every directly surfaced high-value family, A1 must either:
  (a) extract it to a materially useful depth (operative burdens, clause walk,
      full table rows, note structure), OR
  (b) mark it explicitly partial, with:
        - exact reason the family was not fully extracted
        - exact source page range where the family lives
        - what specifically would be required to complete it
"Partial without reason" and "partial without page range" are not allowed.

### Rule 2 — `02_reader_report.md` mandatory subsections
In addition to narrative passes A–G, `02_reader_report.md` MUST include
dedicated subsections, with operative burden summaries (not labels), for each
of the following families when directly surfaced:
  - Item 2 leadership / officers / directors roster — every named person with
    role and tenure as surfaced
  - Items 9, 10, 11, 12, 13, 14, 15, 16 — ONE subsection per item, each with an
    operative burden summary (what the franchisee must do, what the franchisor
    controls, what the franchisee is forbidden from doing)
  - Item 18 public figures / endorsements / limitations — every named figure,
    compensation if disclosed, restriction scope
  - Item 20 — EVERY directly surfaced table, not just headline totals
  - Item 21 — auditor opinion + statement families + every directly surfaced
    note family (walked, not just listed)
  - Key exhibits — franchise agreement, lease, new-term, growth, loan docs,
    franchisee list, state addenda — each with an operative burden summary,
    NOT just an exhibit label

### Rule 3 — `03_tables.json` Item 20 completeness
Every directly surfaced Item 20 table must be extracted as a first-class table
object. A1 may NOT stop at Table 1 or at a single "sample" table when Tables
2, 3, 4, 5, or more are directly surfaced. "Tables 2–5 not extracted because
Table 1 covered the headline" is NOT acceptable.

### Rule 4 — `04_exhibits.json` required fields
For every material exhibit (franchise agreement, lease, new-term addendum,
growth / development / area-development addendum, loan docs, franchisee list,
state addenda, and any other exhibit with operative burden), the exhibit
object MUST include these fields in addition to the existing catalog fields:
  - clause_walk_status          ("complete" | "partial" | "labeled_only")
  - material_clause_count        integer — number of operative clauses walked
  - operative_terms_extracted    array of operative term names actually pulled
  - why_not_complete             string — required whenever clause_walk_status
                                 is not "complete"; must cite reason and pages
"labeled_only" is permitted ONLY with a populated why_not_complete and a
matching retry task in `07_retry_tasks.md`.

### Rule 5 — `05_canonical.json` structured promotions
The canonical JSON must promote structured facts (not prose) for:
  - Item 2 leadership roster        — array of {name, role, tenure, source_pages}
  - Items 9–16 operative burdens     — one structured burden object per item
  - Item 18 public figure restrictions — array of {figure, scope, source_pages}
  - Item 20 — every directly surfaced table as a structured object
  - Item 21 note-family structure    — array of {note_family, walked, source_pages}
  - Key exhibit burdens              — structured burden object per key exhibit
Every promoted structure keeps source_section, source_pages, confidence.

### Rule 6 — `06_coverage_audit.md` covered-complete gating
A family may NOT be marked covered-complete if ANY of the following are true:
  - only headline totals were extracted for a family with sub-tables
  - a directly surfaced Item 20 table was skipped
  - a directly surfaced Item 21 note family was not walked
  - a material exhibit was only labeled and not clause-walked
  - an Item 2 / 9–16 / 18 family was surfaced but left thin
In any of these cases the family must be marked partial with the exact reason.

### Rule 7 — `07_retry_tasks.md` mandatory retries
A1 must emit a targeted retry task (EXECUTE by default) for every directly
surfaced instance of:
  - Item 20 table omitted from `03_tables.json`
  - Item 21 note family omitted from the walk
  - Key exhibit with clause_walk_status != "complete"
  - Item 2 leadership roster left thin
  - Items 9–16 operative burden left thin
  - Item 18 public figure / endorsement family left thin
Retries may be marked SKIP only with an explicit, family-specific rationale —
never as a blanket skip.

### Rule 8 — Operating rule: exhaustiveness over compactness
Do NOT optimize for compactness. When in doubt, prefer exhaustive directly
supported structure over compressed summary. Length is not a defect. A thin
write-up of a directly surfaced family IS a defect.

### Rule 9 — Item 6 multi-page fee table
